人物视线随鼠标移动探究

发表于 2020-09-07 分类于 Unity
本文字数： 2.7k 阅读时长 ≈ 2 分钟

动画问题

我想要在untiy里实现人物的头跟随鼠标移动的一种功能，但是我是头一次接触有关人体模型的一些操作，所以感觉走了很多弯路。我最开始想到的办法就是用脚本直接控制头的transform组件，将鼠标的坐标用一定方式映射为头的角度参数，但是这种方法的问题是无法和人物的动画结合起来；另外一个办法是使用Blend-tree，它可以将不同的动画混合起来，真正开始做起来的时候发现这个思路好像有问题…

控制transform

这个思路是很好想到的，主要需要解决的问题有两个，如何将鼠标的坐标转换为3D空间中的坐标和头部的角度控制问题。

开始觉得直接获取鼠标的坐标就可以了，但是让人物的头的方向对着鼠标并不是需要平面坐标系的坐标，所以我需要一个把鼠标坐标转换为空间坐标系的函数，然后我发现Camera对象有一个方法叫ScreenPointToRay通过传入鼠标的坐标可以得到一条视景体的近平面(Near Clip Plane)到远平面(Far Clip Plane)的一个Ray对象，而我们需要的主要是是这个射线的射出点的坐标，在场景中摄像头是直接对着人物模型的，所以最后让模型的头部转向射线的原点即可。 

public class EyeMove : MonoBehaviour
{
    public Camera camera;
    public Transform head;

    void Start()
    {
        head = this.GetComponent<Transform>();
    }

    void Update()
    {
        Ray ray=camera.ScreenPointToRay(Input.mousePosition);

        //Debug.DrawRay(ray.origin, ray.direction * 10, Color.yellow);

        Vector3 relativePos = ray.origin - head.position;

        head.localEulerAngles = new Vector3(relativePos.x*200, 0, relativePos.y*50);

    }
}

我在代码中用的是localEulerAngles，这里先说明一下什么是EulerAngles，使用这种方式设置的角度是非增量形式的，你设置多少就是多少，如果你想使用增量方式设置角度就应该用Transform的Rotate方法。再解释local，local的意义是相对，所以local欧拉角的设置是相对于父物体的，这里其实应该用非local的版本，但是我不知道怎么设置的最后成功了，所以请读者们不要效仿我的代码🐶

Blend-Tree

Blend-Tree是Unity设计的一种将动画混合起来的机制，你可以通过一个或两个参数控制每个单独的动画在最终动画中的一个权重，多用于混合步行和跑步等相似的动作。由于我只是简单探索下这个功能，所以就记录下使用过程中的一些细节。

首先你需要在Animator controller里右键点击Create State->From New Blend Tree，然后点击新出现的BlendTree进入BlendTree界面。

点击中间的Blend Tree，然后在unity右边的Inspector页面可以看到有一个Motion列表，你需要在那里给Blend Tree添加动作，Blend Type决定了你用几个参数控制动作的混合以及参数的精度。参数的添加和给普通的动画控制器添加参数是一样的。在完成一系列操作后如下图。

选择二维混合方式后会出现右边类似雷达图的东西，中心的红点代表你现在的两个参数组成的坐标，两个蓝点分别代表我导入的两个动作。在Blend Tree运行过程中，参数会根据游戏的输入进行变动，在我的例子里就是根据鼠标的坐标进行变动，所以红点也会发生变动，如果红点进入了蓝点的作用范围，那么相应蓝点的动作就会在混合动作中占高的权重。

但是我最终发现这个功能和我想要实现的效果不太一样。最大的问题就是我录制的动画是不停的播放的，也就是说它会不停的摇头，我想实现的效果头部跟随鼠标移动，这需要让Blend Tree里的参数控制还要准确控制动画播放到了第几秒，所以这方法有根本上的问题。但本着探索的精神我还是记录下来我对混合树的理解。

IK系统

最终破案，使用IK系统是这个功能比较好的选择，我之前走了太多弯路了。简单介绍下IK，IK是Inverse Kinematic反向动力学的缩写，与FK相反，IK是子物体带动父物体，说人话就是你移动模型的手等末端模块，胳膊也会跟着移动。在我这个例子里，利用IK可以直接让头部旋转，那么为什么IK系统知道头部在哪呢，实际上这是unity的IK的特别之处。

unity的IK系统是基于Humanoid模型的，非Humanoid模型无法使用，这一点可以看下unity论坛(3L)里的讨论。由于Humanoid模型在创建过程中Unity会对真实的骨骼进行抽象，简单来说就是把头、胳膊、腿等部位和真实的骨骼进行映射，所以Humannoid模型是很简单的，这种特性让动画变得可重用了，在一个模型上设计的动画可以迁移到另一模型上，对于我这种不懂建模动画的来说是很爽了，更详细的说明可以参考这个网址。正是因为IK是基于Humanoid的，所以IK自然知道头在哪个地方，你只要调用相应的函数就行了。

头部的旋转可以用以下几行代码很简单的实现。

 void OnAnimatorIK(int layerIndex)
    {
        animator.SetLookAtPosition(ray.origin);
        animator.SetLookAtWeight(1);  
    }

注意要让OnAnimatorIK调用需要先让脚本绑定对应的模型，然后打开脚本绑定模型的动画控制器中的IK Pass，如图

SetLookAtPosition设定了头的方向，除此之外还需要使用另一个方法SetLookAtWeight，该方法设置的是IK会多大程度干涉正在运行的动画，也就是IK运动的权重。 

public void SetLookAtWeight(float weight, float bodyWeight = 0.0f, float headWeight = 1.0f, float eyesWeight = 0.0f, float clampWeight = 0.5f);

weight表示全局参数，是后面所有参数的系数。后面的body、head、eys分别代表身体各部位参与LookAt过程的权重，如果你想让头部旋转的过程让身体也稍微跟着动一动，那么你就可以把body的权重调高一些。最后一个参数我还不太明白它的作用，关于IK更详细的说明参考这个博客。

插入图片

在markdown中插入图片不是件简单的事情，你需要准备图片的路径，可以是本地的也可以是网络的，然后在markdown中引用它，问题在于如何管理这些图片，如果把它们放在一个地方比较乱，所以hexo提供了一个插件来简化这件事情，hexo-asset-image，安装方法如下。 

npm install https://github.com/CodeFalling/hexo-asset-image --save

然后把站点配置文件中的post_asset_folder设置为true即可。需要注意的是hexo官网在资源文件夹中说明了这一方法，但是它只说明了config文件中的修改，如果没有安装插件直接做了修改的情况下，单独浏览文章时图片是能正常显示的，但是在主页图片无法正常显示，因为在开启post_asset_folder后是用相对路径引用图片。

网络阻塞问题

在unity进行网络操作时，一般会涉及IO问题，比如发送一个Http请求，按照http协议的规定，发送完后还会有一个response包，所以在等待的过程中，程序会一直卡在这个操作上，从游戏的角度来看，整个游戏会卡住，给用户带来极其糟糕的体验。我通过调查，得知了两个或许解决这个问题的方案，协程和多线程。

这两个方案都可以产生有多个线程的效果，实际上都是假的多线程，它们频繁的切换主线程和子线程来达到并行执行的假象，为了达到这种假象，主线程和子线程的操作都会被割裂成很多块，而它们区别便可以在这里看出来，多线程的操作是在系统层面分片的，也就是说，它破坏了代码的原子性，玩不好会出现问题；而协程则是在更大粒度上上的分片，这个分片粒度是用户可以自己操作的；还有一点需要注意的是协程如果堵塞，那么主线程也会堵塞，所以我们的网络IO问题是不能用协程的，不过我还没有搞懂什么时候使用协程,但是可以确定的是，多线程可以用在需要复杂算法、逻辑处理、IO处理的地方。关于协程的资料可以看下这些资料。

滚动条问题

在进行聊天时，因为聊天消息很多所以一定会涉及滚动条的操作，但是Unity的原生GUI还是要编写一些代码才能使滚动条的长度响应内容的变化的，这里需要说明一下Unity现在有三套UI系统，一种是引擎原生的UGUI，另一种是有人自己写的框架NGUI，还有一种是在调试中使用的GUI，它需要在OnGUI函数中直接写出UI代码。这里我并没有做深入的探索，因为要让滚动条响应内容变化很简单。

简单介绍下Unity原生的滚动部件，你可以在UI部件中看到Scroll View和Scroll bar，其中bar是一个滑动条，是View中的一个部件，通过创建View你可以直接创建有关滑动窗口的一整套组件，如果你更想自定义你的滑动UI，那可以不用这个View，自己搭一个。


上图的左边是View组件的外观，右边为View的层次结构。刚建好的View是同时有水平滚动条和垂直滚动条的，需要根据你的需求删除或者通过代码操作。View里面有两个基本部件，ViewPort和ScrollBar，ViewPort是我们的可见区域，它起到了一个mask的作用，起到只展示Content一部分的作用；Scrollbar是滚动条组件，Area我不太明白有什么作用，Handle就是你要拖动的那个条。

content里要放你要展示的内容物，在我的应用里我放的是代表对话的一个个Text，在聊天过程中，每次发送消息都会克隆Text，但是只是在Content里加入内容物，滚动条的长度不会响应并发生变化，关键点是要改变Content的大小，在每次加入新Text后我会用代码修改Content中Rect Transform组件的Height属性，这样滚动条便会有响应，需要注意的是，Height属性并不能直接改变，你需要访问该组件的sizeDelta属性才能改变它，它是一个二维向量。还有一个小细节是如果你想改变滚动条对鼠标滚轮的敏感程度在Scroll View里有一个Scroll Sensitivity。

Unity开发笔记

调试技巧

附加到Unity

开始开发的时候没有调试功能还可以接受，但是等到程序规模变大了之后不能调试真的是很难受了，除了下面说的单元测试外还有一种方式可以在运行过程中调试，这种方式就是我们在打开VS时总能看见的附加到进程。简单介绍下该功能，附加到进程是一种调试方案，你在VS里打开的代码可能只是主程序的一个dll代码，而这种方式可以让你在主程序运行时打一个断点去调试那些dll代码。

在Unity里主程序是你的游戏，你在播放时游戏便运行了。你可以在你的脚本里打上断点，然后在游戏执行相关的操作便可以开始调试，这种方法一般是主程序是个大程序，无法频繁编译，或者你没有主程序的代码，它在别的机器上。

一开始我的VS里没有附加到Unity，所以我当时缺了Visual Studio Tools for Unity，你需要在VS的关于窗口里进行VS的更新，更新时注意把Unity编辑器的勾打掉，因为你已经安装了。安装后重启Unity和VS便可看到“附加到Unity”选项。

单元测试

在untiy中使用单元测试的一个原因是，VS本身不支持直接调试unity代码，所以你无法执行这些脚本代码。但是使用单元测试，你可以以程序的最小单元，去执行测试，在这里呢，最小单元是指函数，你可以针对一个函数创建一个单元测试，然后就可以运行那个函数的代码了。

同时我也去查了一下单元测试的本来用法，单元测试是在软件工程里的一种调试的办法，一般来说它测试的对象都是函数，你可以想想在一个大型项目中函数的数量是很庞大的，你需要通过测试函数的稳定性来确保整个项目的稳定性，单元测试可以做到这一点，有人研究过，执行单元测试的团队会在debug方面少付出很多时间。
